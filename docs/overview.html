ejs-sales-agents-poc2 · project overview

A config-driven outreach factory on AWS AgentCore

One versioned YAML file describes how a sales account becomes ten ready-to-review outreach artifacts. An engine interprets that file inside Bedrock AgentCore — fetching the live web, scoring the buying committee deterministically, and writing every step down once so a re-run costs nothing.

6 phases · all shipped 92 offline tests deployed on 296497502276 / us-east-1 replay 0.8s / account ~$0.32 / account cold

What it does

The user summary

A BDR team hands the system a batch of target accounts. For each one it: finds the right three people to contact (the HRIS buying committee — champion, economic buyer, technical evaluator), verifies they're reachable, researches the company, and drafts a personal email, LinkedIn note, and call script for each contact — nine artifacts plus an account brief, queued for human review. Nothing is ever sent automatically.

Two things make this build different from its predecessor (poc1):

Behavior lives in config, not code

Every stage — its model tier, prompt file, policy, and order — is declared in pipelines/bdr_outreach.yaml. Swapping the contact-selection rule or pinning the web-fetch source is a YAML edit. A validator refuses bad configs at startup, including a lint that keeps the expensive Opus tier out of non-generation stages.

AgentCore-native, end to end

Runtime hosts the engine; the Browser tool fetches the live web (no third-party search API); Memory gives each BDR their own writing voice; Gateway exposes internal tools over MCP; Observability prices every stage of every run.

The identical engine and config run on a laptop with zero AWS: in-memory state, local output folder, fixture pages, static voice. Environment selects the backends; the code never branches.

Functional flow

The life of one account

Real numbers from the verified Meridian Robotics runs. Every solid dot is a checkpoint — written once to DynamoDB, replayed for free forever. Dashed dots are idempotent writes that re-run on purpose.

  1. validate haiku1.3s · $0.0013 · parallel with identify

    Required-field check is computed in code — the model only writes advisory reasons (it was caught hallucinating field presence in both directions). Meridian: VALID.

  2. identify · fetch_pages toolfallback chain: attached → browser → fixture

    The AgentCore Browser session hunts team/leadership pages, runs "<company> <role> LinkedIn" searches per committee role, and a funding/hiring signals pass. Raw page text checkpoints — a replay never opens a browser. (Basecamp, a real account with no fixture: 8 pages fetched.)

  3. identify · enrich sonnet4.8s · $0.014 · the one model call in identification

    Judges only what code can't: anchor quality, roles, LinkedIn presence — strictly from the fetched pages. Empty pages → empty result, no model call, nothing invented.

  4. identify · prioritize policy11ms · $0 · byte-identical port from poc1

    The deterministic committee engine: extraction → 0–100 Contact Access score → P1–P3. Meridian: 68/B — CPO, CFO, IT Director; the IT Director's email is composed from a HIGH-confidence firstname.lastname pattern (guessed addresses attach only at high confidence — that's policy, in config).

  5. identify · persist_identified writes identified_contacts.json now — survives any later failure
  6. verify × 7 haiku17.5s threaded · $0.015 · one checkpoint per contact

    4 CRM contacts + 3 identified. VERIFIED means a valid email or phone. The junk "Dead Lead" fails; the pattern-guessed P3 passes.

  7. reconcile → barrier policy9ms · alphabetical select-3, swappable in config

    Selection: Alvarez, Berg, Krishnan — the identified IT Director makes the final three. Then the declarative gate: require: [account_valid, three_verified].

  8. research + summary sonnet19s · $0.022 · provenance-tagged, exactly 5 bullets

    Company facts and trigger events, each origin-tagged. The 5-bullet brief is the shared context every generator reuses — and "exactly 5" is enforced in code.

  9. generate × 9 opus61s threaded · $0.27 — the cost center, checkpointed per artifact

    3 contacts × (email + LinkedIn + call script). Each prompt carries the BDR's voice from Memory — retrieved by bdr_id, static snippet as fallback. Checkpoint keys like gen#c-102#email mean a crashed batch only re-pays for what never finished.

  10. persist idempotent by key · re-runs on every replay · terminal: ARTIFACTS_QUEUED_REVIEW

    Writes the summary, 9 artifacts, manifest, and cost trace to S3 in poc1's exact key layout, then appends the account's event to Memory ("batch ran, 10 artifacts queued, access 68/B").

The example batch (mocks/sample_batch_mixed.json) sends three accounts with deliberately different starting points — rich CRM data, thin CRM data, and none at all — through the deployed Step Functions envelope. All three reached ARTIFACTS_QUEUED_REVIEW with 10 artifacts each.

Unplanned bonus: a real mid-run fault

Harborline's first runtime invocation died with a transient RuntimeClientError. Step Functions retried; the second attempt served the 20 finished checkpoints from DynamoDB and computed only the 2 that never completed — manifest: computed=2 cached=20, done in 8.8s, zero duplicate Opus spend. The recovery design, exercised by a genuine production fault.

Operations

Running & moving it

  • Install anywhere: edit config.json (region, resource prefix, model IDs) and run python scripts/install.py — it preflights model access and AgentCore service availability before creating anything, then deploys the whole stack. Verified by a foreign-prefix dry run that re-derived all 16 resource names from config alone.
  • Run a batch: python scripts/run_batch.py --batch-id b1 — a fresh id is a cold run; the same id is the free replay. --fixture-only pins deterministic demo data.
  • Inspect cost: python scripts/cost_report.py b1 bdr-emea-07 acct-001.
  • Local, zero AWS: python -m poc2.run pipelines/demo.yaml; the full test suite (pytest -q, 92 tests) never touches the network.
  • Tear down: python scripts/uninstall.py removes everything the installer created. Idle cost of the deployed stack is ~nil (no always-on compute).
